Phase 0: Discovery & Ideation

A tool-agnostic, specification-driven architecture for AI-native software development that prioritizes understanding before building
💡 Specifications as Executable Artifacts • 🤝 Human-AI Collaboration • 📊 Context Engineering

Core Philosophy: Spec-First, Tool-Agnostic

Phase 0 transforms discovery from an ad-hoc process into a systematic, repeatable practice. Instead of jumping to solutions, we invest deeply in understanding the problem space through structured conversations, validated documentation, and strategic context management.

🎯 Specifications as Executable Artifacts

Requirements documents become the "source of truth" that drive all downstream work. The specification itself becomes executable—directly generating implementations rather than just guiding them.

🤝 Human-AI Collaboration

Agents facilitate structured discovery through multi-turn conversations with validation loops, not one-shot generations. The human remains in control, with AI amplifying their ability to extract and synthesize information.

🧠 Context Engineering

Strategic management of information flow maintains focus and prevents context collapse. Agents receive exactly the right context at the right time—no more, no less.

🔄 Incremental Refinement

Multi-phase approach over single-pass attempts. Each iteration adds clarity without discarding valuable insights from previous conversations.

✅ Validation Before Implementation

Built-in checkpoints and review cycles ensure completeness before moving to the next phase. Problems caught in discovery cost hours, not weeks.

🛠️ Tool Agnostic

This strategy works with or without code because it focuses on structured thinking and documentation. Can be enhanced programmatically, but doesn't require it.

⚠️ Why This Matters
Most AI-assisted development fails not because of model limitations, but because teams don't provide adequate context or skip foundational discovery. This architecture ensures AI agents have the structured information they need to be genuinely helpful.

Three-Layer Framework

The framework combines Markdown prompt engineering, agentic primitives, and context engineering to transform ad-hoc AI experimentation into reliable, repeatable processes.

1 Markdown Prompt Engineering
Structured prompts using Markdown (headers, lists, links) guide AI reasoning and produce predictable, consistent outputs. The hierarchical structure of Markdown naturally organizes complex information and makes it easier for both humans and AI to parse.
Application in Discovery:
  • Templated markdown documents for interviews, requirements, and analysis
  • Headers create logical separation of concerns (Problem | Solution | Constraints)
  • Lists and tables organize stakeholder feedback systematically
  • Reduced ambiguity through consistent formatting
2 Agentic Primitives
Reusable, configurable building blocks (like .instructions.md, .prompt.md, .spec.md, .memory.md) that encapsulate roles, rules, and workflows. These primitives document both successes and failures, creating an evolving knowledge base.
Discovery Phase Primitives:
  • .discovery-facilitator.instructions.md - Interview protocols and frameworks
  • .problem.spec.md - Problem statement templates
  • .feasibility.memory.md - Accumulated technical knowledge
  • .stakeholder.context.md - Persistent stakeholder information
3 Context Engineering
Strategic management of contextual information to optimize AI performance within memory constraints. Just like humans, LLMs have finite memory—providing the right context (not just more context) is critical for effectiveness.
Techniques for Discovery:
  • Session Splitting: Separate interviews for different stakeholder groups
  • Modular Instructions: Targeted questioning based on role (technical vs. business)
  • Memory Files: Persistent knowledge across discovery sessions
  • Selective Context: Only load relevant prior discoveries for each conversation
💡 Key Insight
These three layers work together synergistically. Markdown provides structure, primitives provide reusability, and context engineering provides focus. Remove any layer, and reliability degrades significantly.

Agent Architecture

Stage 1: Discovery Facilitator

Conducts structured interviews using systematic frameworks (Five Whys, Jobs-to-be-Done) to extract and validate requirements without bias toward solutions.

🎤 Structured Interviewing

Generate contextually-aware questions that adapt based on stakeholder responses, identifying gaps and ambiguities in real-time.

👂 Active Listening

Paraphrase stakeholder statements for validation, extract implicit requirements, and flag contradictions between different stakeholders.

📝 Documentation Generation

Create problem statements, success criteria matrices, and constraint documents with clear rationales and traceability.

## .discovery-facilitator.instructions.md ### Role You are an expert business analyst conducting discovery interviews to uncover the true problem space before any solutions are discussed. ### Question Generation Framework #### Five Whys Protocol For each stated problem: - Ask "why is this a problem?" five levels deep - Document each layer separately - Identify root causes vs symptoms #### Jobs-to-be-Done Framework For each user goal: - What job are they trying to accomplish? - What is their current solution? - What are the pain points? - What would success look like? ### Output Artifacts 1. Interview Transcript (raw conversation) 2. Problem Statement (synthesized from transcript) 3. Success Criteria (measurable outcomes) 4. Constraints Document (technical, business, regulatory)

Stage 2: Feasibility Analyzer

Evaluates technical viability, risks, and technology fit without prescribing specific solutions. Focuses on what's possible, not what tools to use.

🔍 Technology Landscape

Research current approaches to similar problems, identify proven patterns and anti-patterns, evaluate maturity of potential solutions.

⚠️ Risk Identification

Analyze technical, business, and external risks with likelihood and impact assessments, plus mitigation strategies.

✓ Constraint Validation

Verify stated constraints are real vs assumed, identify hidden constraints, document tradeoffs clearly.

Aspect Discovery Facilitator Feasibility Analyzer
Primary Focus Understanding the problem space Evaluating solution viability
Stakeholder Interaction High - continuous conversation Medium - targeted validation
Research Depth Domain and business context Technical precedents and risks
Output Type Qualitative requirements Quantitative assessments
Decision Authority None - facilitates understanding Advisory - provides recommendations

Discovery Workflow

The discovery workflow follows a structured progression from initial problem articulation through validated feasibility assessment, with multiple validation loops ensuring completeness.

1

Initial Problem Articulation

Stakeholder provides high-level description of the problem or opportunity. Discovery Facilitator begins structured questioning using Five Whys to move from symptoms to root causes.

📄 Initial Problem Description 🎯 Stakeholder Context
2

Multi-Stakeholder Interviews

Conduct separate sessions with different stakeholder groups (executives, technical teams, end users). Each session uses role-appropriate questioning and builds on prior interviews through memory files.

📋 Interview Transcripts 🔄 Cross-Reference Notes ⚠️ Contradiction Log
3

Problem Statement Synthesis

Synthesize all interview findings into a clear, validated problem statement with quantified business impact, identified root causes, and measurable success criteria. Use incremental delta updates to preserve detail.

📊 Problem Statement v1.0 💰 Impact Quantification 🎯 Success Criteria
4

Stakeholder Validation Loop

Present synthesized problem statement to all stakeholder groups for confirmation. Resolve any contradictions or misunderstandings before proceeding to feasibility analysis.

✅ Approval Matrix 🔄 Revision Log
5

Technical Feasibility Research

Feasibility Analyzer researches similar implementations, identifies proven patterns, assesses technical risks, and evaluates organizational capability gaps. No technology prescriptions—only viability assessment.

🔬 Case Study Analysis ⚠️ Risk Register 📈 Capability Gap Analysis
6

Feasibility Recommendation

Generate comprehensive feasibility report with Go/No-Go/Conditional recommendation. Include confidence level, supporting evidence, and mitigation strategies for identified risks.

📋 Feasibility Report 🎲 Confidence Score 🛡️ Mitigation Plans
7

Phase 0 Completion Validation

Review all artifacts against completion checklist. Ensure problem is fully understood, feasibility is assessed, and team is aligned before transitioning to specification phase.

✅ Completion Checklist 📦 Artifact Package ➡️ Phase 1 Handoff
🔄 Iterative Nature
This workflow is rarely linear. New information from feasibility analysis may trigger additional stakeholder interviews. Contradictions discovered in validation may require revisiting earlier steps. The key is maintaining traceability through all iterations.

Context Engineering Strategies

Effective context engineering ensures AI agents have exactly the information they need—no more, no less. This prevents context overload, maintains focus, and avoids the common pitfall of "context collapse" where important details are lost over time.

Memory Management

Separate persistent context (always loaded) from session-specific context (loaded only when relevant). This reduces token usage and improves response quality by eliminating noise.

## .discovery.memory.md ### Persistent Context (loaded in all discovery sessions) - Organization: Acme Corp - Industry: SaaS B2B - Team Size: 50 engineers - Current Tech Stack: .NET, React, PostgreSQL - Previous Projects: [list with outcomes] ### Session-Specific Context (loaded only when relevant) - Stakeholder Role: Support Team Lead - Prior Interviews: CEO (003), CTO (005) - Related Projects: Customer Portal Redesign (2024)

Session Splitting

Use distinct agent sessions for different stakeholder groups and analysis types. Each session maintains its own context, preventing cross-contamination and overload.

Discovery Phase Sessions:
  • Executive Interviews: Business goals, strategic constraints, success criteria
  • Technical Interviews: Current system limitations, integration needs, technical debt
  • User Interviews: Actual usage patterns, workarounds, feature desires
  • Synthesis Session: Combine findings, resolve conflicts, generate artifacts

Avoiding Context Collapse

Context collapse occurs when iterative rewriting loses crucial details over time. Solution: Use incremental delta updates rather than monolithic rewrites.

## Version 1: Initial problem statement "Support team is overwhelmed" ## Version 2: After CEO interview (DELTA) "Support team is overwhelmed" + Root cause: 60% of tickets are repetitive FAQs + Business impact: $200K annual revenue loss from churn ## Version 3: After Support Lead interview (DELTA) "Support team is overwhelmed" + Root cause: 60% of tickets are repetitive FAQs + Business impact: $200K annual revenue loss from churn + Team morale impact: 40% turnover in last 12 months + Current workaround: FAQ page that customers don't find
🧠 Context Engineering Benefits
  • Reduced Token Usage: Only relevant context loaded, lowering costs
  • Improved Accuracy: Less noise means better signal detection
  • Faster Response: Smaller context windows = faster processing
  • Better Reasoning: Agents can focus on what matters
  • Preserved History: Delta updates maintain full audit trail

Implementation Patterns

This architecture can be implemented in multiple ways, from pure conversation-based workflows to fully automated systems. The choice depends on your team's maturity, tooling, and needs.

Pattern Best For Complexity Automation Level
Conversational Small teams, rapid iteration, exploratory projects Low Manual with AI facilitation
Template-Based Standardized processes, compliance requirements Medium Semi-automated artifact generation
Programmatic Enterprise scale, complex integrations, analytics High Fully automated with human checkpoints

Pattern 1: Conversational (Tool-Agnostic)

Pure conversation between human and AI agent following structured protocols. All artifacts created manually in markdown files with AI assistance for synthesis and validation.

Workflow:
  1. Human initiates discovery conversation with agent
  2. Agent follows .instructions.md protocol for questioning
  3. Human and agent collaboratively create problem-statement.md
  4. Agent validates completeness using checklist
  5. Human approves and moves to feasibility phase

Tools Needed: Any AI chat interface (Claude, ChatGPT, etc.) + text editor

Pattern 2: Template-Based (Semi-Automated)

Uses spec-kit style slash commands and templates to scaffold artifacts. AI generates initial drafts from templates, human refines through iterative conversation.

# Example slash command workflow /discovery.initialize "E-commerce checkout optimization" # Creates .discovery folder with templates /discovery.interview role="CEO" framework="five-whys" # Conducts structured interview, saves transcript /discovery.synthesize # Generates problem statement from all interviews /discovery.validate # Checks completeness against checklist

Tools Needed: Claude Code, Gemini CLI, or GitHub Copilot + spec-kit templates

Pattern 3: Programmatic (Advanced)

Full implementation with validation pipelines, automated traceability, and integration with project management systems. Suitable for enterprise-scale operations.

public class DiscoveryOrchestrator { public async Task<DiscoveryPackage> ExecuteDiscoveryPhase( ProjectInitiation initiation) { // 1. Conduct multi-stakeholder interviews var interviews = await ConductInterviews( initiation.Stakeholders ); // 2. Synthesize problem statement with delta tracking var problemStatement = await SynthesizeProblemStatement( interviews, useDeltaUpdates: true ); // 3. Validate with stakeholders var validation = await ValidateWithStakeholders( problemStatement ); // 4. Conduct feasibility analysis var feasibility = await AnalyzeFeasibility( problemStatement, initiation.Constraints ); // 5. Generate traceability matrix var traceability = await GenerateTraceability( interviews, problemStatement, feasibility ); return new DiscoveryPackage { ProblemStatement = problemStatement, FeasibilityReport = feasibility, Interviews = interviews, TraceabilityMatrix = traceability, CompletionStatus = await ValidateCompleteness() }; } }

Tools Needed: Custom C# application + LLM API + project management integration

Phase 0 Completion Validation

Problem Definition: Problem statement is clear, specific, and traces back to stakeholder interviews
Root Cause Analysis: Root causes identified through Five Whys (not just symptoms)
Business Impact: Impact is quantified with specific metrics (revenue, cost, time, etc.)
Success Criteria: Measurable success criteria defined with clear acceptance thresholds
Constraints: All constraints validated as real (not assumed) with documented rationale
Feasibility: Technical viability assessed with confidence level and risk mitigation plans
Stakeholder Alignment: All stakeholder groups reviewed and approved findings
Documentation: All artifacts complete, traceable, and free of contradictions
Handoff Readiness: Package ready for Phase 1 (Specification) with no open questions
🎯 Success Metric
A successful Phase 0 completion means Phase 1 (Specification) proceeds without needing to revisit discovery. If you find yourself going back to "clarify the problem," Phase 0 wasn't complete. The goal is one-pass implementation success through comprehensive context.
